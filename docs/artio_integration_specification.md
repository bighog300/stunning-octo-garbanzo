# Artio Integration Specification

## 1. Purpose

This document defines how the Artio app integrates with the Scrapy + Airflow + dbt + Superset data pipeline.

The pipeline produces clean, normalized art records. Artio consumes those records, presents them to users, and stores app-specific decisions such as approvals, rejections, enrichment, and public visibility.

The recommended integration flow is:

```text
analytics marts / app views → Artio backend API → Artio UI
```

---

## 2. Integration Goals

Artio should be able to:

- Read clean artwork records generated by dbt.
- Search, filter, and browse collected artwork records.
- Display source attribution and provenance.
- Support human review before records are made public.
- Approve, reject, or enrich records.
- Track review status separately from raw scraped data.
- Avoid direct dependency on crawler internals.
- Continue operating even if a crawl or dbt run fails.

---

## 3. Integration Non-goals

The Artio app should not:

- Run crawlers directly.
- Write to the `raw` schema.
- Modify dbt-managed `analytics` tables.
- Depend on Scrapy spider implementation details.
- Treat scraped data as automatically approved public content.
- Download or host images without a separate image/compliance policy.

---

## 4. Data Boundary

The pipeline and app should have clear boundaries.

```text
raw schema       → owned by Scrapy/Airflow
analytics schema → owned by dbt
app schema       → owned by Artio
```

Artio may read from:

```text
analytics.mart_artworks
analytics.mart_artists
analytics.mart_sources
app.artwork_records
app.review_queue
app.approved_artworks
app.rejected_artworks
```

Artio may write to:

```text
app.review_queue
app.approved_artworks
app.rejected_artworks
app.record_enrichments
```

Artio should not write to:

```text
raw.*
analytics.*
```

---

## 5. Recommended Access Pattern

Recommended architecture:

```text
PostgreSQL analytics/app schemas
        ↓
Artio backend service
        ↓
Artio frontend/UI
```

Do not connect the frontend directly to the database.

The backend should own:

- Query logic
- Authentication and authorization
- Review state changes
- Search/filter API
- Record enrichment API
- Public visibility logic

## 6. Database Views for Artio

## 6.1 app.artwork_records

Primary Artio-facing view.

Purpose:

- Combine dbt-cleaned artwork data with review state.
- Provide one stable read model for Artio.

Expected fields:

```text
artwork_id
raw_artwork_id
artist_id
artist_name
artwork_title
year_start
year_end
artwork_date_text
medium_text
medium_category
dimensions_text
height_cm
width_cm
depth_cm
price_text
price_numeric
currency_code
source_name
source_domain
source_url
image_url
thumbnail_url
description
quality_score
duplicate_group_key
is_duplicate_candidate
review_status
public_visibility
crawl_timestamp
created_at
```

Recommended view logic:

```sql
CREATE OR REPLACE VIEW app.artwork_records AS
SELECT
    a.artwork_id,
    a.raw_artwork_id,
    a.artist_id,
    a.artist_name,
    a.artwork_title,
    a.year_start,
    a.year_end,
    a.artwork_date_text,
    a.medium_text,
    a.medium_category,
    a.dimensions_text,
    a.height_cm,
    a.width_cm,
    a.depth_cm,
    a.price_text,
    a.price_numeric,
    a.currency_code,
    a.source_name,
    a.source_domain,
    a.source_url,
    a.image_url,
    a.thumbnail_url,
    a.description,
    a.quality_score,
    a.duplicate_group_key,
    a.is_duplicate_candidate,
    COALESCE(rq.review_status, 'pending') AS review_status,
    COALESCE(aa.public_visibility, false) AS public_visibility,
    a.crawl_timestamp,
    a.created_at
FROM analytics.mart_artworks a
LEFT JOIN app.review_queue rq
    ON a.artwork_id = rq.artwork_id
LEFT JOIN app.approved_artworks aa
    ON a.artwork_id = aa.artwork_id;
```

## 7. Artio App Tables

## 7.1 app.review_queue

Stores review state for records.

```sql
CREATE TABLE app.review_queue (
    id UUID PRIMARY KEY,
    artwork_id UUID NOT NULL,
    review_status TEXT NOT NULL DEFAULT 'pending',
    priority TEXT DEFAULT 'normal',
    assigned_to TEXT,
    review_notes TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    reviewed_at TIMESTAMPTZ,
    reviewed_by TEXT
);
```

### review_status values

```text
pending
approved
rejected
needs_enrichment
duplicate
```

---

## 7.2 app.approved_artworks

Stores approval and visibility state.

```sql
CREATE TABLE app.approved_artworks (
    id UUID PRIMARY KEY,
    artwork_id UUID NOT NULL,
    approved_by TEXT,
    approved_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    public_visibility BOOLEAN NOT NULL DEFAULT false,
    notes TEXT
);
```

---

## 7.3 app.rejected_artworks

Stores rejected records.

```sql
CREATE TABLE app.rejected_artworks (
    id UUID PRIMARY KEY,
    artwork_id UUID NOT NULL,
    rejection_reason TEXT NOT NULL,
    rejected_by TEXT,
    rejected_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    notes TEXT
);
```

### rejection_reason values

```text
duplicate
missing_required_data
bad_source
copyright_concern
not_relevant
low_quality
other
```

---

## 7.4 app.record_enrichments

Stores human edits or enrichment data without modifying dbt marts.

```sql
CREATE TABLE app.record_enrichments (
    id UUID PRIMARY KEY,
    artwork_id UUID NOT NULL,
    enriched_field TEXT NOT NULL,
    original_value TEXT,
    enriched_value TEXT NOT NULL,
    enriched_by TEXT,
    enriched_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    notes TEXT
);
```

Purpose:

- Preserve raw and dbt-generated data.
- Store human-curated improvements separately.
- Allow future versioning and rollback.

## 9. Review Workflow

Recommended workflow:

```text
New dbt mart record
   ↓
Appears in app.artwork_records with review_status = pending
   ↓
Reviewer opens record in Artio
   ↓
Reviewer chooses approve / reject / needs enrichment / duplicate
   ↓
Artio writes review state to app schema
   ↓
Record becomes available or hidden based on state
```

---

## 10. Public Visibility Rules

A record should be public only if:

```text
review_status = approved
public_visibility = true
source attribution exists
source_url exists
no unresolved copyright/compliance flag exists
```

For MVP, default should be:

```text
public_visibility = false
```

Approval should not automatically make a record public unless explicitly requested.

---

## 11. Search and Filtering

Artio should support search across:

```text
artist_name
artwork_title
medium_text
description
source_name
```

Recommended PostgreSQL search index:

```sql
CREATE INDEX idx_artwork_records_search
ON analytics.mart_artworks
USING GIN (
  to_tsvector(
    'english',
    coalesce(artist_name, '') || ' ' ||
    coalesce(artwork_title, '') || ' ' ||
    coalesce(medium_text, '') || ' ' ||
    coalesce(description, '')
  )
);
```

Search behavior:

```text
Exact title match should rank high
Artist match should rank high
Source/domain filters should be fast
Pending review records should be easy to isolate
```

## 13. Duplicate Handling

dbt identifies duplicate candidates.

Artio handles final decisions.

Duplicate-related fields:

```text
duplicate_group_key
is_duplicate_candidate
```

Recommended duplicate review behavior:

```text
Show records in same duplicate_group_key together
Allow reviewer to mark one as canonical
Allow reviewer to reject duplicates
Preserve source URLs for all duplicate records
```

Future app table:

```sql
CREATE TABLE app.canonical_artworks (
    id UUID PRIMARY KEY,
    canonical_artwork_id UUID NOT NULL,
    duplicate_artwork_id UUID NOT NULL,
    decided_by TEXT,
    decided_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    notes TEXT
);
```

---

## 14. Enrichment Handling

Human enrichment should overlay dbt data in the app layer.

Example:

```text
analytics.mart_artworks.artist_name = Van Gogh
app.record_enrichments.enriched_value = Vincent van Gogh
```

Artio can display:

```text
Preferred value: Vincent van Gogh
Source value: Van Gogh
```

This preserves provenance.

---

## 15. Image Handling

For MVP:

```text
Store image URLs only
Display remote image thumbnails where permitted
Do not permanently store image files
Do not make image-heavy public pages without compliance review
```

Artio should handle broken images gracefully:

```text
Show placeholder image
Flag missing/broken image
Allow reviewer to continue
```

Future image fields:

```text
image_storage_url
image_license_status
image_checked_at
image_is_available
```

## 16. Permissions and Roles

Suggested Artio roles:

```text
admin
reviewer
viewer
public_user
```

### admin

Can:

```text
View all records
Approve/reject records
Assign review tasks
Change public visibility
Manage sources
```

### reviewer

Can:

```text
View review queue
Approve/reject records
Add enrichment notes
Mark duplicates
```

### viewer

Can:

```text
View internal records
Filter/search data
Cannot approve/reject
```

### public_user

Can:

```text
View only approved + public records
```

## 17. Error Handling

API should return clear errors.

Examples:

```text
404 artwork not found
400 invalid review_status
400 invalid rejection_reason
409 record already approved/rejected
403 insufficient permissions
500 database error
```

Approval conflict example:

```text
If a record is already rejected, approving it should require explicit override.
```

## 21. Acceptance Criteria

The Artio integration is ready when:

```text
[ ] Artio can read app.artwork_records
[ ] Artio can list artwork records
[ ] Artio can filter by review_status
[ ] Artio can open artwork detail records
[ ] Artio displays source attribution
[ ] Artio can approve records
[ ] Artio can reject records
[ ] Approval/rejection writes to app schema
[ ] Public records are hidden unless explicitly made public
[ ] Artio does not write to raw or analytics schemas
```
